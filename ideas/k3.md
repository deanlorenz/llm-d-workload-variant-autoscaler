# ✅ **Section: Analyzer K3 — Concurrency‑Based Supply–Demand Modeling for Workload Types**

## 1. Preliminaries

We consider a distributed inference system (llm‑d) serving requests of **workload types**

$$
w \in \mathcal{W}
$$

where each workload type is characterized by:

*   $IL(w)$: expected input length in tokens
*   $OL(w)$: expected output length in tokens
*   $KV(w)$: expected KV‑cache load per request
*   $E2E(w,v)$: expected end‑to‑end latency of workload type $$w$$ processed on backend variant $$v$$

Backend variants $$v \in \mathcal{V}$$ represent different vLLM backend configurations (different models, quantization, page size, GPU type, max batch size, etc.).

The llm‑d cache‑aware scheduler is **w‑aware** but **v‑oblivious**, meaning routing is not conditioned on backend type.  
WVA is **v‑aware**, and this difference is exploited to compute supply/demand per variant.

***

# ✅ 2. Observable Quantities

At runtime, llm‑d exposes:

*   $$N(w)$$: number of concurrent requests of workload type $$w$$ across the entire inference pool
*   $$N(v)$$: number of concurrent requests served by backend variant $$v$$
*   $$N(w,v)$$: estimated concurrency of workload type $$w$$ on variant $$v$$

Since routing is w‑oblivious with respect to variants, we derive:

$$
N(w,v) \approx N(v) \cdot \frac{N(w)}{\sum_{w'} N(w')}
$$

which is a proportional allocation consistent with the observed fan‑out of requests.

This gives the **effective per‑variant workload distribution** without modifying llm‑d router behavior.

***

# ✅ 3. KV‑cache Load Model per Workload Type

For each workload type $$w$$, we define expected KV load:

$$
KV(w) = E[H\%] \cdot (IL(w) + 0.5 \cdot OL(w))
$$

where:

*   $$H\%$$ is the observed KV hit rate
*   The 0.5 factor reflects the incremental KV growth during decoding
*   This is consistent with llm‑d’s K1/K2 semantics but **conditioned on workload type**.

A variant $$v$$ has total available KV:

$$
KV_{\max}(v)
$$

Thus the **maximal concurrency** of workload type $$w$$ on variant $$v$$ is:

$$
N_{\max}(w,v) = \frac{KV_{\max}(v)}{KV(w)}
$$

This represents the concurrency threshold beyond which variant $$v$$ becomes KV‑saturated for workload $$w$$.

***

# ✅ 4. End‑to‑End Latency Model

For each $$(w,v)$$:

$$
E2E(w,v) \approx TTFT(w,v) + ITL(w,v) \cdot OL(w)
$$

We assume:

*   TTFT is dominated by prefill processing of IL(w) and GPU saturation effects
*   ITL grows approximately linearly with KV utilization
    $$
    ITL(w,v) \approx a_v \cdot KV\%(v) + b_v
    $$
    where $$a_v, b_v$$ are learned per variant.

Thus:

$$
E2E(w,v) \approx TTFT(w,v) +
OL(w) \cdot \left(a_v \cdot \frac{N(w,v)\cdot KV(w)}{KV_{\max}(v)} + b_v\right)
$$

This is the **variant‑specific latency curve** as concurrency changes.

***

# ✅ 5. K3 Supply Function

The goal of Analyzer K3 is to quantify **how much concurrency a single instance of backend v can sustainably serve**, defined as:

$$
S_v = \sum_{w \in \mathcal{W}} \frac{N_{\max}(w,v)}{E2E(w,v)}
$$

Interpretation:

*   For each workload type $$w$$, the “service capacity” is proportional to  
    $$\frac{1}{E2E(w,v)}$$.
*   $$N_{\max}(w,v)$$ is the concurrency ceiling for that workload on that backend.
*   K3 adds all these contributions into a single **per‑backend supply metric**.

This creates a normalized quantity comparable across variants.

***

# ✅ 6. K3 Demand Function

Observed instantaneous demand is:

$$
D_v = \sum_{w \in \mathcal{W}} \frac{N(w,v)}{E2E(w,v)}
$$

This encodes:

*   Observed concurrency of type $$w$$ on variant $$v$$
*   Divided by the time each request type consumes resources on that variant

This is a **true concurrency‑normalized demand signal**.

***

# ✅ 7. Scale Decision Rule

Analyzer K3 recommends scaling variant $$v$$ by:

$$
\Delta_v = \left\lceil \frac{D_v}{S_v} \right\rceil - I_v
$$

where:

*   $$I_v$$ is the number of currently running instances of variant $$v$$
*   $$\Delta_v > 0$$ → scale out
*   $$\Delta_v < 0$$ → scale in
*   $$\Delta_v = 0$$ → no change

This rule naturally balances:

*   workload type distributions
*   backend variant differences
*   KV limits
*   latency model
*   concurrency observations

No queueing assumptions are required; the metric is purely resource‑based.

***

# ✅ 8. Summary of K3

Analyzer **K3** introduces a concurrency‑versus‑latency scaling signal:

*   **Supply** is characterized by $$N_{\max}(w,v)$$ and $$E2E(w,v)$$.
*   **Demand** is the observed runtime concurrency $$N(w,v)$$ normalized by latency.
*   The ratio $$D_v / S_v$$ gives a unified autoscaling load index.

This formulation complements K1 (KV‑based) and K2 (GPU‑based) analyzers and captures scenarios where **workload composition**, **variant differences**, and **latency behavior** must jointly determine the correct autoscaling action.
